# 여권 OCR & 쇼핑 면세 혜택 (Passport OCR & Tax-Free)

> 상태: **계획(예정)** — 착수 선행조건: **My 메뉴 화면 디자인 수정 완료**
> 관련 백로그: BACKLOG `#26`

## 1. 목표

My 메뉴에서 사용자가 **본인 여권을 촬영 → 여권 정보 추출·저장**하면, 외국인 **쇼핑 면세(Tax Refund) 혜택**을 제공한다. 방한 외국인의 핵심 니즈(면세 쇼핑)를 앱 안에서 연결한다.

## 2. 사용자 플로우

1. My 탭 → "여권 등록 / 면세 혜택" 진입 (진입점 위치·디자인은 My 디자인 수정 후 확정)
2. 여권 사진 촬영 또는 갤러리 선택 — 카메라/사진 권한은 just-in-time 요청, 거부 시 수동 입력 degrade
3. OCR(MRZ 파싱)로 정보 추출: 성/이름, 여권번호, 국적, 생년월일, 만료일
4. 사용자가 추출 결과 확인·수정 후 저장 (체크 디지트 검증)
5. **면세 혜택 카드 활성화**: 면세 한도·즉시환급/사후환급 안내, (Phase 2) 제휴 매장 면세 QR

## 3. 기술 설계

- **OCR/MRZ**: 여권 하단 MRZ 2줄(ICAO 9303 TD3) 파싱. 기존 `supabase/functions/ocr`(Google Vision, 키 설정됨) 확장 또는 MRZ 전용 파서 추가. MRZ 체크 디지트로 정확도 검증, 실패 시 수동 입력.
- **저장 위치 (결정 필요)**:
  - 옵션 A — 로컬 전용(MMKV, 암호화): 서버 미전송, 면세 안내는 클라이언트. 개인정보 리스크 최소.
  - 옵션 B — 서버(Supabase): 본인만 접근 RLS + 민감 필드 암호화 + 명시 동의 + 삭제 기능. 크로스기기·검증 연계 유리.
  - 권장: **최소 수집** 원칙(면세에 필요한 필드만), 여권 원본 이미지는 저장하지 않음.
- **면세 로직**: 한국 외국인 Tax Refund 규정(즉시환급 한도·사후환급 절차). 정확성은 세무/법무 검토 필요. 제휴 매장 연계는 Phase 2.

## 4. 개인정보·보안 (중요 — PLANNING §12 원칙 강화)

- 수집 **목적·보관기간·삭제방법 명시 동의** 화면 선행
- 여권 **이미지는 보관하지 않고 추출 텍스트만**(또는 마스킹), 가능하면 온디바이스 처리
- 서버 저장 시: 민감 필드 암호화 + RLS(본인만) + 사용자 삭제 기능
- 게스트도 가능하되 로그인 권장(linkIdentity로 데이터 승계 — voice/gganbu 세션과 동일 원리)

## 5. 데이터 모델 (옵션 B 채택 시)

```
passports: id, user_id(→users.id, RLS 본인만), surname, given_names,
           passport_no(암호화), nationality, birth_date, expiry_date,
           verified, created_at, updated_at
```

## 6. 화면

- **My 탭**: "여권 등록 / 면세 혜택" 진입 항목 (위치·디자인 = My 디자인 수정 후 확정)
- `app/passport.tsx`(모달 또는 탭 내 화면): 촬영 → 검토/수정 → 저장 → 면세 혜택 카드

## 7. 단계

- **Phase 1**: 촬영·OCR(mock-first)·MRZ 파싱·정보 확인 UI·로컬 저장
- **Phase 2**: 면세 혜택 카드/안내, (택1) 서버 저장(동의·암호화·RLS·삭제)
- **Phase 3**: 제휴 매장 면세 QR·환급 연계

## 8. 선행조건 / 의존성

- **착수 조건: My 메뉴 화면 디자인 수정 완료** (진입점·레이아웃 확정 후 시작)
- 카메라/사진 권한: `expo-image-picker`(구성됨)
- OCR: `ocr` Edge Function + Google Vision 키(설정됨)

## 9. 결정 필요 (착수 전)

- 저장 위치: 로컬 전용 vs 서버 — 개인정보 정책 확정
- 면세 규정 정확성(세무/법무 검토), 제휴 매장 범위·연계 방식
